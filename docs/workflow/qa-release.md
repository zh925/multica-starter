# QA 与发布

## Staging QA

输入必须包含确认需求 commit、PR/head SHA、测试环境、build ID、影响范围和验收标准。当前 CI/CD 未接入时，测试环境和构建来源必须人工记录并核实。

QA 结论只能是：

- `PASS`：允许进入 UAT。
- `FAIL`：存在阻断问题，回流开发。
- `CONDITIONAL PASS`：仅表示有待人工接受的遗留风险，不能自动进入发布。

## 缺陷等级

- `Blocker`：禁止上线。
- `Critical`：核心功能或重大安全问题，必须修复。
- `Major`：重要功能错误，通常必须修复。
- `Minor`：可由用户决定延期。
- `Suggestion`：改进建议，不自动阻断。

每个独立缺陷创建独立 Issue。修复后必须重新经过代码评审和受影响的 QA。

## 发布前置条件

- QA PASS。
- 用户对指定版本 UAT 通过。
- 明确的生产发布批准和窗口。
- 数据库/基础设施影响、备份、迁移和回滚方案已审核。
- 发布清单、负责人和验证范围完整。

## 发布后验证

检查服务健康、核心路径、错误率、延迟、资源、日志、告警、关键依赖和业务指标。异常由人类负责人决定回滚或 Hotfix。只有观察窗口通过后，父需求才能关闭。

## CI/CD 接入要求

后续接入时至少提供：

- PR 检查和测试结果；
- Staging 部署成功事件；
- Production 部署成功事件；
- 部署事件中的 environment、commit SHA、build/deployment ID 和 URL；
- Secret 管理、审批、回滚和事件幂等策略。
